32: Versie Controle met Git

Dit is een (aangepaste) vertaling van de Git Tutorial van Jae Woo Lee and Stephen A. Edwards Columbia University (maart 2013)

Git is een versie controle systeem voor programmacode. Zo'n systeem is nuttig als je in een team werkt, maar ook als je alleen werkt. Het is een handig hulpmiddel om bij te houden welke veranderingen je hebt aangebracht aan je programmacode en om als dat nodig is terug te stappen naar een vorige versie. Als je in een team werkt kun je met een versie controle systeem bijhouden wie welke bijdragen heeft gedaan aan de code, waar conflicten zitten in de code en je kunt verschillende ontwikkellijnen opzetten en daar moeiteloos tussen switchen. Overigens: je kunt git ook gebruiken voor versiecontrole van willekeurig welke type bestanden. Git werkt met 'binary objects' en dat kan een html pagina zijn, maar ook een word document of een afbeelding!

Git kun je hier downloaden: https://git-scm.com/. Er zijn ook portable versies (die vanaf een USB stick draaien) en er zijn versies voor alle grote besturingssystemen.

De basisversie van Git is een programma dat alleen via de commandline werkt. Je kunt zowel in de cmd (of powershell) van windows werken als in een speciale (bij Git 'meegeleverde') bash-omgeving, dat is een onder Linux veel gebruikte commandline. Behalve deze commandline versie zijn er voor Git ook windowed clients. In diverse editors (Sublime, Brackets e.d.) zitten Git clients ingebouwd of zijn er als extensie (add on) aan toe te voegen.

In dit document gebruiken we verder de command line in Linux of Windows

Heel belangrijke sites in de Git-wereld (behalve git-scm.com) zijn Github en Bitbucket. Hier kun je als persoon of als team git projecten hosten en van allerlei documentatie over git vinden. Van harte aanbevolen. Met name voor het werken in teams zijn Github en Bitbucket perfecte oplossingen. Zolang je project 'Open-Source' is zijn Github en Bitbucket gratis oplossingen.

Neem even de tijd voor het downloaden en installeren van jouw git versie.

Configureren van je Git-omgeving

Vertel git je naam en emailadres:

git config --global user.name "Jouw volledige naam
git config --global user.email "jouw_adres@ergens.domein";

Git slaat dit op in de git configuratie file: ~/.gitconfig file. Het is een goed idee om de EDITOR omgevingsvariabele van Git te zetten. Dit bepaalt de editor die wordt gestart voor het typen van log berichten.

Concept: drie 'toestanden'

Git is een systeem dat je toe past op een map waarin je wilt gaan werken. Als je wilt gaan programmeren in de map project1 kun je daar een nieuwe Git map van maken. Hoe dat gaat zie je in de volgende paragraaf. Vanaf het moment dat je dat gedaan hebt zal Git alle bestanden in die map (en submappen) indelen in drie 'toestanden':

  • Working directory: dat zijn alle bestanden in de map en submappen, belangrijk of niet.(1)
  • Staging area: dat zijn alle bestanden in de map en submappen die je aan wilt gaan melden voor versiecontrole
  • Repository: dat zijn alle bestanden die daadwerkelijk zijn aangemeld bij de versiecontrole
(1) Op verzoek kan Git bepaalde bestanden negeren

De belangrijkste Git commando's zijn de commando's die te maken hebben met deze toestanden of de overgangen ertussen:

  • git status : om overzicht te krijgen over de drie toestanden
  • git add : om (veranderde) bestanden van de working directory naar de staging area te krijgen
  • git commit : om (veranderde) bestanden van de staging area over te brengen naar de repository
  • git checkout: om (veranderde) bestanden van de repository over te brengen naar de working directory
